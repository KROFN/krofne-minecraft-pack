# RELEASE_FLOW.md — Процесс обновления модпака

Документ описывает ежедневный процесс обновления модпака через GitHub для владельца,
использующего **krofnePackUpdater** в Admin mode.

---

## Структура GitHub-репозитория

Репозиторий `KROFN/krofne-minecraft-pack` (ветка `main`) должен содержать:

```
krofne-minecraft-pack/
├── manifest.json          ← главный файл, который скачивает приложение
└── mods/                  ← все .jar файлы модпака
    ├── jei-1.20.1-forge-15.2.0.27.jar
    ├── appleskin-forge-mc1.20.1-2.4.3.jar
    └── ...
```

⚠️ Каждый мод из `manifest.json` должен иметь `downloadUrl`, ведущий на raw-файл в этом
репозитории. Формат URL:

```
https://raw.githubusercontent.com/KROFN/krofne-minecraft-pack/main/mods/{fileName}
```

---

## Ежедневное обновление модпака через GitHub

### 1. Добавление нового .jar

1. Скачай или собери `.jar` файл мода (убедись, что это **Forge 1.20.1**).
2. Положи `.jar` в свою **локальную рабочую папку модов** — именно её ты укажешь в Admin mode.
3. Удали старую версию того же мода из папки (если обновляешь), чтобы при сканировании
   не было дубликатов.

### 2. Удаление старого .jar

1. Просто удали `.jar` из локальной рабочей папки модов.
2. При следующем сканировании (Scan) он не появится в списке.
3. ⚠️ Не забудь также удалить файл из GitHub-репозитория — иначе он будет «висеть» мёртвым
   грузом и занимать место (приложение его не скачает, но порядок в репозитории важен).

### 3. Генерация manifest.draft.json через Admin mode

1. Открой приложение **krofnePackUpdater**.
2. Перейди в **Settings** → включи **Admin mode**.
3. Перейди на вкладку **Admin**.
4. Выбери **папку с модами** (ту, где лежат все нужные `.jar`).
5. Нажми **Scan** — приложение найдёт все `.jar` и вычислит SHA-512 для каждого.
6. Заполни поля:
   - **packName** — название модпака (например, `krofnePack`).
   - **packVersion** — новая версия (например, `1.5.0`), **обязательно увеличь** относительно
     предыдущей.
   - **minecraftVersion** — `1.20.1`.
   - **loader** — `forge`.
   - **loaderVersion** — версия Forge (например, `47.2.0`).
   - **githubRawBaseUrl** — `https://raw.githubusercontent.com/KROFN/krofne-minecraft-pack/main/mods`
7. Укажи **oldManifestUrl** — `https://raw.githubusercontent.com/KROFN/krofne-minecraft-pack/main/manifest.json`.
   Это позволит приложению:
   - Сохранить метаданные модов, которые не изменились (по совпадению SHA-512).
   - Сохранить `allowedExtraMods` из старого манифеста.
8. Отредактируй **changelog** (см. шаг 5 ниже).
9. Нажми **Generate** — приложение создаст manifest JSON.
10. Нажми **Preview** — проверь содержимое манифеста.
11. Нажми **Save** — файл сохранится как `manifest.draft.json` в выбранной локации.

### 4. Проверка downloadUrl

Для каждого мода в сгенерированном манифесте `downloadUrl` формируется автоматически:

```
{githubRawBaseUrl}/{fileName}
```

Например:
```
https://raw.githubusercontent.com/KROFN/krofne-minecraft-pack/main/mods/jei-1.20.1-forge-15.2.0.27.jar
```

✅ **Обязательно проверь**, что каждый URL реально отдаёт файл. Открой несколько ссылок
в браузере — должен начаться скачивание `.jar`, а не ошибка 404.

⚠️ Если в `githubRawBaseUrl` есть лишний `/` в конце — он автоматически обрезается,
но лучше писать без него.

### 5. Обновление packVersion

- Увеличивай `packVersion` при **каждом** изменении состава модпака.
- Рекомендуемый формат: `MAJOR.MINOR.PATCH` (семантическое версионирование).
  - Добавлен новый мод → `MINOR` (1.4.0 → 1.5.0).
  - Обновлён существующий мод → `PATCH` или `MINOR` (1.4.0 → 1.4.1).
  - Удалён мод → `MINOR` (1.4.0 → 1.5.0).
- ⚠️ Если не изменить `packVersion`, пользователи не поймут, что модпак обновился.

### 6. Заполнение changelog

В разделе **Changelog Editor** добавь новую запись:

```json
{
  "version": "1.5.0",
  "date": "2025-03-04",
  "items": [
    "Добавлен мод X",
    "Обновлён мод Y до версии 2.0",
    "Удалён мод Z (конфликт)"
  ]
}
```

Правила:
- `version` должен совпадать с `packVersion`.
- `date` — дата релиза в формате `YYYY-MM-DD`.
- `items` — понятные описания изменений для игроков.
- Старые записи changelog **не удаляй** — они показывают историю.

### 7. Заливка на GitHub

1. Переименуй `manifest.draft.json` → `manifest.json`.
2. Скопируй **все** `.jar` файлы из рабочей папки в локальный клон репозитория
   (в папку `mods/`).
3. В терминале, из корня репозитория:

```bash
# Добавить все изменения
git add .

# Закоммитить с понятным сообщением
git commit -m "Release v1.5.0: добавлен X, обновлён Y, удалён Z"

# Запушить
git push origin main
```

⚠️ **Порядок важен**: и `manifest.json`, и все `.jar` файлы должны быть в одном коммите.
Если залить манифест без `.jar`, пользователи получат ошибку скачивания.

✅ После пуша подожди 10–30 секунд — GitHub CDN должен обновить raw-ссылки.
Проверь: открой `https://raw.githubusercontent.com/KROFN/krofne-minecraft-pack/main/manifest.json`
в браузере и убедись, что видишь новый `packVersion`.

### 8. Тестирование на локальной копии mods

1. Создай **тестовую папку** `mods` (пустую или с предыдущей версией модпака).
2. Открой приложение в обычном (не Admin) режиме.
3. Укажи тестовую папку `mods`.
4. Нажми **Check** — приложение скачает манифест, просканирует папку и построит sync plan.
5. Проверь: количество установленных / отсутствующих / лишних модов должно быть корректным.
6. Нажми **Sync** — приложение скачает недостающие, заменит устаревшие, переместит лишние
   в папку `_disabled_by_krofne_pack`.
7. ✅ Проверь, что в тестовой папке `mods` теперь ровно те `.jar`, что указаны в манифесте.
8. Запусти Minecraft с тестовой папкой — убедись, что всё запускается без крашей.

### 9. Когда можно отдавать друзьям

✅ Модпак можно отдавать, если:

- [ ] Тестовая синхронизация прошла без ошибок.
- [ ] Minecraft запускается и заходит на сервер.
- [ ] В приложении после **Check** отображается: все моды `installed`, `0 missing`, `0 wrong version`.
- [ ] `manifest.json` и все `.jar` доступны по raw-URL (нет 404).
- [ ] `packVersion` обновлён.

Теперь можно отправить друзьям ссылку на приложение:
`https://github.com/KROFN/krofnePackUpdater/releases/latest`

## Частые ошибки

### ❌ Забыл залить .jar на GitHub

**Симптом**: друг нажимает Sync, скачивание зависает или падает с ошибкой.

**Решение**: проверь, что каждый `.jar` из манифеста доступен по `downloadUrl`.
Открой URL в браузере — должен начаться скачивание файла.

### ❌ SHA-512 не совпадает

**Симптом**: приложение показывает «wrong version» для мода, который только что скачан.

**Причина**: файл на GitHub побился при загрузке, или ты перезалил `.jar` без перегенерации
манифеста.

**Решение**:
1. Перегенерируй манифест (Scan → Generate) — SHA-512 пересчитается из текущих файлов.
2. Залей новый `manifest.json` вместе с `.jar`.

⚠️ SHA-512 — 128 шестнадцатеричных символов. Если длина не 128 — что-то сломано.

### ❌ Raw URL неправильный

**Симптом**: 404 при скачивании или загрузке манифеста.

**Правильный формат**:
```
https://raw.githubusercontent.com/{owner}/{repo}/{branch}/{path}
```

**Частые ошибки**:
- Использован `github.com` вместо `raw.githubusercontent.com`.
- Опечатка в имени репозитория или ветки.
- Файл залит в другую ветку (не `main`).
- Файл лежит не в той папке (`mods/` vs корень).

### ❌ packVersion не обновил

**Симптом**: друг говорит «у меня та же версия», хотя моды изменились.

**Решение**: всегда увеличивай `packVersion` при любом изменении состава модпака.

### ❌ Друг выбрал не ту папку mods

**Симптом**: приложение показывает «0 installed» и хочет скачать всё заново,
или находит десятки лишних модов.

**Решение**: правильный путь к папке модов:
```
%APPDATA%\.minecraft\mods          (Windows, стандартный лаунчер)
```

⚠️ Не путай с `.minecraft/mods` внутри другого экземпляра Prism/ATLauncher.
Приложение попытается автоопределить путь, но лучше проверить вручную.

### ❌ GitHub отдаёт 404

**Симптом**: приложение не может скачать манифест.

**Причины**:
1. Репозиторий приватный — raw-ссылки не работают без авторизации.
2. Файл ещё не попал на CDN — подожди 30 секунд после пуша.
3. Опечатка в URL манифеста. Стандартный:
   ```
   https://raw.githubusercontent.com/KROFN/krofne-minecraft-pack/main/manifest.json
   ```
4. Файл называется `manifest.draft.json`, а не `manifest.json` — не забудь переименовать.

## Версионирование манифеста (Best Practices)

### Формат packVersion

Используй семантическое версионирование: `MAJOR.MINOR.PATCH`

| Изменение | Инкремент | Пример |
|-----------|-----------|--------|
| Добавлен мод | MINOR | 1.4.0 → 1.5.0 |
| Обновлён мод | MINOR или PATCH | 1.4.0 → 1.4.1 |
| Удалён мод | MINOR | 1.4.0 → 1.5.0 |
| Переработка состава (3+ изменений) | MINOR | 1.4.0 → 1.5.0 |
| Критическое изменение (новый Minecraft/loader) | MAJOR | 1.5.0 → 2.0.0 |

### schemaVersion

Текущий `schemaVersion: 1`. Не меняй его, пока не выходит обновление самого приложения
с новой структурой манифеста.

### manifestUpdatedAt

Генерируется автоматически при Generate — ISO 8601 дата.
Не редактируй вручную.

### Правило одного коммита

⚠️ **Всегда** коммить `manifest.json` и все новые/обновлённые `.jar` **одним коммитом**.
Это гарантирует атомарность: либо всё обновилось, либо ничего.

```bash
# ✅ Правильно — всё в одном коммите
git add manifest.json mods/
git commit -m "Release v1.5.0"
git push origin main

# ❌ Неправильно — манифест и .jar в разных коммитах
git add manifest.json
git commit -m "Update manifest"
git push origin main
# ... через час ...
git add mods/
git commit -m "Add jars"
git push origin main
```

### oldManifestUrl — зачем и как

При генерации нового манифеста укажи `oldManifestUrl`:

```
https://raw.githubusercontent.com/KROFN/krofne-minecraft-pack/main/manifest.json
```

Это позволит:
- Сохранить `id`, `name`, `side`, `required` для модов, чей SHA-512 совпал
  (мод не изменился — metadata переносится).
- Сохранить `allowedExtraMods` из старого манифеста.
- Автоматически переименовать мод, если имя файла изменилось при том же хеше.

✅ Если не указать `oldManifestUrl`, новые моды получат `id` из имени файла,
`side: "both"` и `required: true` — придётся править вручную.
